# 实用功能建议与 PRD 合集（MoSCoW）

## 0. 用户画像与场景假设

以项目现有功能（个人/团队空间、一次性链接、审计、资产归档、导入导出）推断主要用户分层：

- **个人用户**：少量密钥、偶尔导入/导出、担心丢失与安全泄露。
- **小团队/值班**：需要共享只读访问（一次性链接）、需要审计与快速失效。
- **管理员/Owner**：更关注治理：资产归档、风险面板、提醒与审计追踪。

## 1. MoSCoW 优先级清单

| 优先级 | 功能 | 预期收益指标（示例） | 实现成本（相对） |
|---|---|---:|---:|
| Must | 审计/分享链接“到期与风险提醒”（站内+邮件） | 降低“遗留有效链接”数量 | 中 |
| Must | 导出/导入安全与可恢复性增强（密钥轮换/备份策略提示） | 降低数据不可恢复事故 | 低-中 |
| Should | 一次性链接“即将过期/剩余次数=1”快捷筛选 + 批量就地更新 | 缩短排查与治理时间 | 中 |
| Should | 团队资产归档“责任人/观察者”通知与 SLA | 提升归档完成率 | 中 |
| Could | PWA 离线页/本地只读模式（不含密钥明文） | 更好移动端体验 | 中-高 |
| Won’t（本阶段） | 小程序/H5 双端深度差异化 | 成本较高且现阶段价值不明确 | 高 |

## 2. PRD-01：分享链接风险提醒（Must）

### 背景

一次性访问链接可减少共享风险，但现实中仍会出现：

- 有效链接长期遗留未撤销
- 创建人离职/邮箱缺失导致无法触达
- 团队管理员难以持续跟踪“高风险状态”

### 目标

- 支持按规则自动识别风险链接并提醒创建人/管理员处理。
- 提供可追踪的提醒审计记录。

### 用户故事

- 作为团队管理员，我希望每天看到“仍有效且被访问过的链接”列表，并可一键提醒创建人撤销。
- 作为链接创建人，我希望在链接长时间有效时收到提醒，以免遗忘。

### 验收标准

- 支持配置规则（默认）：
  - 仍有效且距离过期 > X（例如 1 小时）并且 `view_count>0`
  - 或仍有效且创建超过 N 天（例如 7 天）
- 支持通知渠道：
  - 站内提示（dashboard/团队工作台风险卡）
  - 邮件（仅对已绑定邮箱用户）
- 每次提醒写入审计（TeamAudit/TOTPEntryAudit 任选其一，但必须可追踪）

### 原型（文字）

- 团队工作台 “安全风险概览” 增加“需提醒”计数与入口。
- 一次性链接团队审计页增加快捷筛选“需提醒”，并支持批量提醒（现已有批量提醒，可在此基础上增加“需提醒”筛选与规则展示）。

### API 定义（建议）

- `POST /totp/teams/<id>/share/one-time/remind/`（已存在批量 remind）
  - 入参：`ids[]`
  - 输出：`ok, requested, reminded, skipped_no_email`
- 新增（可选）：
  - `GET /totp/teams/<id>/share/one-time/risk/` 返回规则命中的 link id 列表（用于前端展示）

### 灰度发布

- 默认仅站内提示；邮件提醒需要显式开关（settings/env）。
- 先对管理员可见；再逐步开放给普通创建人自查。

### 回滚策略

- 关闭定时任务/开关即可停止提醒，不影响既有链接与审计数据。

## 3. PRD-02：一次性链接审计“就地更新”（Should）

### 背景

当前批量失效/提醒完成后整页 reload，容易：

- 丢失滚动位置与筛选状态
- 在移动端回到顶部，效率较低

### 目标

- 批量操作后无需整页刷新：只更新被处理行的 status 与按钮可用性，并更新顶部计数。

### 验收标准

- 操作成功后：
  - 将被处理的 active 行标记为 revoked（前端层即可，不强依赖后端返回每个 id 的状态）
  - 取消其勾选、禁用单条“失效”按钮
  - 更新 “已选 X，可失效 Y” 与 “当前有效 N” 的展示

### 技术方案

- 批量 API 目前返回 `updated/requested`，可扩展返回 `updated_ids`（可选）。
- 若不扩展 API：前端根据所选 active 复选框集合做就地更新即可。

## 4. PRD-03：资产归档 SLA 与通知（Should）

### 背景

团队资产归档依赖人推动，常见痛点：

- 未归属长期堆积
- 责任人不清晰、提醒链条断

### 目标

- 对“未归属条目”建立 SLA（例如 7 天内归档）
- 支持对资产 owners/watchers 的通知（站内/邮件）

### 验收标准

- 资产详情页展示：
  - 未归属条目数、最久未归属时长
  - owners/watchers 可配置
- 每周自动发送一封“归档待办”汇总（可开关）

### 数据与迁移（可能需要）

- 若要做 SLA 统计，建议记录“归属变更时间”（可在 `TOTPEntryAudit` 里已有，但查询成本高；可选加字段 `asset_assigned_at`）。

## 5. PRD-04：备份与密钥轮换可视化（Must/Should）

### 背景

项目依赖 `TOTP_ENC_KEYS` 进行密钥加密与轮换；配置错误是生产事故高发点。

### 目标

- 在管理员可见的“系统状态”页显示：
  - 是否配置 `TOTP_ENC_KEYS`
  - 当前使用的 key 数量
  - 建议轮换策略与演练入口

### 验收标准

- 生产模式未配置时给出明确告警（不一定阻断启动）
- 提供“轮换操作手册”与 checklist（文档即可）

## 6. 对现有架构影响评估

- 以上建议均可在现有 Django 单体结构内落地：
  - 通知：复用现有 email 发送能力（若项目已有邮件配置；当前代码中团队提醒已发送邮件，说明已有发送路径）
  - 定时任务：建议引入 `django-q`/`celery` 或使用系统 cron + management command
  - 数据迁移：仅 PRD-03/04 可能需要新增字段或配置校验

## 7. 迭代看板建议（示例）

- Sprint 1：PRD-01（站内提示 + 筛选 + 审计）
- Sprint 2：PRD-01（邮件提醒 + 灰度开关）+ PRD-02（就地更新）
- Sprint 3：PRD-03（资产 SLA）+ PRD-04（系统状态/轮换可视化）
